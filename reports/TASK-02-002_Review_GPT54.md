## 한 줄 결론
**통합 테스트를 추가했다는 점은 인정되지만, 현재 보고서만 기준으로 보면 “설계대로 잘 되었는지”를 입증하는 수준까지는 아직 못 갔다.**

판정: **조건부 합격**

- 테스트 3개가 모두 “시스템이 대체로 연결되어 동작한다”는 신호는 준다.
- 하지만 각 테스트의 검증 강도가 약해서, **설계 계약이 정확히 지켜졌는지**까지는 충분히 증명하지 못한다.
- 따라서 이번 태스크는 **통합 테스트의 시작점으로는 적절**하지만, **통합 안정성 확보 완료**로 보기에는 이르다.

## 잘 된 점

### 1) 단위 테스트를 넘어서 시스템 간 연결을 보기 시작한 점은 좋다
이번 태스크는 Grid/Transport, Agent/Schedule, SimClock/EventBus처럼 서로 다른 시스템 경계를 한 번에 건드린다. 이 자체는 Sprint 2 초반에 꼭 필요한 방향이다.

### 2) 프로덕션 API를 따라간 점은 긍정적이다
보고서에서 `getAllElevators()`가 스냅샷이 아니라 `std::vector<EntityId>`를 반환하기 때문에, `getElevatorState()`로 상태를 확인했다고 적었다. 이 접근 자체는 캡슐화를 유지하는 쪽이라 좋다.

다만 여기에는 작은 경고도 있다. `VSE_Design_Spec.md` 기준으로도 원래 설계는 `getAllElevators() -> std::vector<EntityId>` + `getElevatorState(id)` 조합이다. 즉 이번 수정은 “실제 API에 맞춘 우회”라기보다, **권위 문서 기준으로는 원래 설계에 맞춘 사용**에 더 가깝다. 보수적으로 보면, 보고서 문구에는 **사양 이해 드리프트**가 조금 남아 있다.

### 3) production code를 건드리지 않고 test-only 범위에서 끝낸 점은 안전하다
새 파일 1개와 CMake 수정만으로 끝낸 점은 좋다. 통합 테스트 태스크에서 production code까지 같이 만지기 시작하면, 테스트 도입이 아니라 기능 수정으로 번지기 쉽다.

### 4) SimClock + EventBus 연결을 숫자로라도 확인한 점은 의미가 있다
`TickAdvanced`, `HourChanged`를 카운터로 확인한 것은 최소한의 안전망으로는 쓸모가 있다. 이벤트가 아예 안 나오거나 flush가 동작하지 않는 큰 문제는 잡을 수 있다.

## 핵심 우려점

## 1. 가장 큰 문제: 테스트들이 “존재 확인” 중심이라 설계 계약 검증으로는 약하다
이번 보고서의 세 테스트는 전부 공통적으로 이런 성격이 있다.

- 어떤 상태가 **언젠가 한 번이라도 나타났는지** 본다.
- 어떤 이벤트 수가 **대략 맞는지** 본다.
- 어떤 시스템이 **완전히 멈추지는 않았는지** 본다.

이런 테스트는 통합 테스트의 1차 방어선으로는 쓸 수 있다. 하지만 보수적으로 보면, 이 정도로는 “설계대로 잘 되었다”라고 말하기 어렵다.

설계 검증에 가까운 통합 테스트라면 아래까지 확인해야 한다.
- 상태 전이 **순서**가 맞는가
- 경계 시점이 **정확한 tick/time**에 맞는가
- 이벤트 **payload**와 **발행 순서**가 맞는가
- 시스템 간 연결이 단순 연결이 아니라 **의도한 계약대로** 움직이는가

현재 보고서는 이 수준까지는 올라오지 못했다.

---

## 2. Elevator 테스트는 “도착했는가”만 보여주고, LOOK/수송 계약은 거의 검증하지 못한다
테스트 1의 이름은 강하다.

`Integration - Elevator: single call, car arrives and opens door`

하지만 실제 assert는 다음 수준이다.
- 최대 200 tick 안에
- 엘리베이터가 floor 3에 도달하고
- 상태가 `DoorOpening`, `Boarding`, `DoorClosing` 중 하나

이 검증으로 확인되는 것은 사실상 이것뿐이다.
- 호출 후 언젠가 3층까지 간다
- 문 관련 상태 비슷한 것이 한 번 나온다

반대로, 아래 중요한 계약은 거의 검증하지 못한다.

### 2-1) LOOK 알고리즘 검증이 아니다
`CLAUDE.md`와 `VSE_Design_Spec.md`는 Phase 1 엘리베이터 알고리즘을 LOOK으로 잡고 있다. 그런데 이번 테스트는 단일 호출 1건만 다룬다.

이 상태에서는 다음을 전혀 확인하지 못한다.
- 같은 방향 호출이 여러 개 쌓일 때 정렬이 맞는가
- 반대 방향 호출이 들어오면 언제 뒤집는가
- call queue가 LOOK 순서대로 유지되는가
- 중간층 정차가 올바른 우선순위를 따르는가

즉, **엘리베이터가 움직인다는 것**은 보이지만, **LOOK가 맞게 구현되었는지**는 증명되지 않는다.

### 2-2) 수송 전체 사이클 검증도 아니다
이 테스트는 “도착 + 문 상태 진입”까지만 본다. 아래는 보지 않는다.
- `ElevatorCalled` / `ElevatorArrived` 이벤트가 발행되는가
- 승객 탑승이 실제로 일어나는가
- passengerCount가 증가/감소하는가
- 탑승 후 목표층으로 이동하는가
- 내린 뒤 queue가 정리되는가
- capacity 제한이 걸리는가

즉, 제목은 수송 시나리오처럼 보이지만 실제로는 **초기 접근 단계만 검사하는 테스트**에 가깝다.

### 2-3) 200 tick 제한은 안전장치이긴 하지만, 계약 검증에는 느슨하다
200 tick 안에 도착한다는 조건은 무한루프 방지에는 도움이 된다. 하지만 보수적으로 보면 너무 넓다.
- 지나치게 느려도 통과할 수 있다.
- 중간에 비정상 왕복을 해도 결국 도착하면 통과할 수 있다.
- 실제 성능/시간 감각이 망가져도 테스트가 초록불일 수 있다.

즉 이 수치는 liveness 보장에는 좋지만, behavior correctness 보장에는 약하다.

## 3. NPC Daily Cycle 테스트는 이름보다 실제 검증 범위가 훨씬 좁다
테스트 2의 이름은 다음이다.

`Integration - NPC: daily schedule transitions (Idle → Working → Resting)`

하지만 실제 assert는 보고서 기준으로 이 정도다.
- 1440 tick (= 1일) 진행
- `Working` 상태가 **어느 시점엔가 한 번** 나타남
- `Resting` 상태가 **어느 시점엔가 한 번** 나타남

이건 통합 테스트의 출발점으로는 가능하지만, 제목이 암시하는 수준의 검증은 아니다.

### 3-1) Idle → Working → Resting “순서”를 검증하지 않는다
현재 설명대로라면 아래처럼 이상한 구현도 통과 가능성이 있다.
- 시작부터 이미 `Working`
- 중간에 잠깐 `Resting`
- 다시 `Working`
- 마지막엔 엉뚱한 상태

즉 상태가 “나타났는지”만 확인하고, **전이 순서와 전이 의미**를 확인하지 않는다.

### 3-2) 설계 문서의 schedule 의미를 충분히 검증하지 않는다
`VSE_Design_Spec.md`에는 `workStartHour`, `workEndHour`, `lunchHour`가 있고, daily cycle은 `home → work → lunch → work → home`으로 설명된다.

그런데 이번 테스트는 아래를 확인하지 않는다.
- 실제 work start가 의도한 시간대에 일어나는가
- lunch 관련 상태/이동이 반영되는가
- work 종료 후 home으로 돌아가는가
- 하루가 바뀌면 다시 cycle이 일관되게 반복되는가

즉 설계 문서에 적힌 “일과 시스템”의 의미를 테스트 제목만큼은 검증하지 못한다.

### 3-3) AgentSystem과 TransportSystem의 실제 연동도 입증하지 못한다
home floor 0, work floor 2이면 층 이동이 필요하므로 수직 이동 수단과의 연동 가능성이 중요하다. 그러나 현재 보고서에는 아래 검증이 없다.
- elevator 호출이 실제로 발생했는가
- waiting / boarding / in-elevator / arrival 과정이 있었는가
- pathfinding과 transport handoff가 실제로 연결되었는가

즉 이번 테스트는 “NPC 상태 머신이 하루 중 변했다”는 수준은 보이지만, **NPC 이동 + 엘리베이터 + 일정 시스템이 진짜 통합되었다**는 증거로는 약하다.

## 4. SimClock + EventBus 테스트는 숫자 카운트만 맞추는 수준이라 이벤트 계약 검증이 부족하다
테스트 3은 다음을 본다.
- 120 tick 진행
- flush each tick
- `tickCount == 120`
- `hourCount == 2`

이 테스트는 최소한의 연결 확인으로는 쓸모가 있다. 하지만 보수적으로 보면 다음 빈칸이 크다.

### 4-1) payload를 검증하지 않는다
`HourChanged`가 2번 왔다고 해서, 그 payload의 hour 값이 1 → 2로 올바르게 증가했는지는 아직 모른다.

### 4-2) 이벤트 순서를 검증하지 않는다
`TickAdvanced`와 `HourChanged`가 같은 tick 경계에서 어떤 순서로 발행되는지가 중요할 수 있다. 지금 테스트는 순서를 보지 않는다.

### 4-3) day boundary를 검증하지 않는다
2시간만 보기 때문에 하루 경계에서 `DayChanged`가 정확히 나오는지, `HourChanged`와 충돌하지 않는지는 여전히 공백이다.

### 4-4) 실제 메인 루프와 동일한 사용 패턴인지 애매하다
보고서에는 “flush after each tick”라고 되어 있다. 이건 테스트 안정성을 위해 흔히 쓰는 방식이지만, 실제 메인 루프에서 `SimClock::update(realDeltaMs)`와 `EventBus::flush()`가 어떤 경계에서 묶여 있는지까지 동일하게 재현하는지는 별도 문제다.

즉 현재 테스트는 EventBus가 완전히 죽어 있지 않다는 것은 보여주지만, **실전 루프에서의 이벤트 계약**을 충분히 증명하지는 못한다.

## 5. Compliance Checklist는 다소 과장되어 있다
보고서는 다음 체크리스트를 모두 충족했다고 적었다.

- No layer boundary violations
- No hardcoded values
- Naming conventions followed
- No exceptions used
- Within Phase 1 scope
- English comments throughout
- File header present

이 중 몇 개는 대체로 수긍 가능하지만, 몇 개는 표현이 너무 강하다.

### 5-1) “No hardcoded values”는 체크리스트 문구가 과하다
보고서 본문만 봐도 테스트 시나리오에는 명시적 상수가 많다.
- floor 0 / floor 3
- shaftX = 5
- capacity = 8
- 200 ticks
- 1440 ticks
- 120 ticks

물론 테스트에서는 이런 고정값이 자연스럽다. 문제는 고정값 자체가 아니라, 체크리스트 문구가 너무 절대적이라는 점이다.

더 정확한 표현은 아래에 가깝다.
- **production balance/config 값은 외부 설정 기반**
- **test fixture 상수는 시나리오 표현을 위해 명시적으로 사용**

보수적으로 보면 현재 체크리스트는 “정책 위반이 없다”기보다 “정책 해석을 넓게 잡았다”에 가깝다.

### 5-2) layer violation 없음은 대체로 맞지만, Bootstrapper 레벨 통합은 아직 아니다
production code를 안 건드렸으니 레이어 침범 가능성은 낮다. 다만 이번 테스트는 여전히 “시스템 조합 테스트”이지, 실제 앱 조립 루트까지 포함한 **상위 통합 테스트**는 아니다.

즉 체크리스트는 틀렸다기보다, **달성 범위를 넓게 보이게 쓰였다**는 느낌이 있다.

## 6. 테스트 수 증가와 실제 안정성 증가는 같은 말이 아니다
보고서는 `3 new + 147 existing = 150/150 passed`를 강조한다. 이 숫자는 분명 의미가 있다. 하지만 보수적으로 보면, 여기서 생기기 쉬운 착시는 다음이다.

- 테스트 수가 늘었다 = 설계 정합성이 높아졌다
- 150/150 통과 = 통합 리스크가 충분히 줄었다

이번 추가 3개는 **있어야 하는 테스트**인 것은 맞다. 다만 그 강도는 아직 약하다. 따라서 이 숫자를 “안정성 확보”로 읽으면 위험하다.

정확한 해석은 이쪽에 가깝다.
- **기존 단위 테스트 위에 최소한의 통합 연기 감지 장치를 얹었다**
- **하지만 설계 계약 검증 테스트로 보기에는 아직 이르다**

## 가장 우려되는 시나리오
이번 보고서 기준으로 가장 우려되는 것은 다음 유형의 버그다.

### 1) 엘리베이터가 결국은 움직이지만, LOOK 우선순위가 틀린 경우
단일 호출 테스트는 통과하지만, 실제 게임에서는 다중 호출 시 비효율적 왕복이나 잘못된 정차 순서가 나올 수 있다.

### 2) NPC가 하루 중 Working/Resting을 둘 다 찍지만, 시간표와 이동 흐름이 틀린 경우
즉 제목은 일과 시스템인데, 실제론 상태 플래그만 바뀌는 수준일 수 있다.

### 3) 이벤트 카운트는 맞지만, 이벤트 시점/순서/payload가 틀린 경우
이 경우 단일 테스트는 초록불이지만, 상위 시스템 연결 시 미묘한 버그가 터질 수 있다.

이 세 가지는 전부 **현재 테스트로는 놓칠 수 있는 유형**이다.

## 보수적 권고

### 1) Elevator 테스트를 “전체 수송 사이클” 기준으로 강화하는 편이 좋다
최소한 아래 중 일부는 추가되는 편이 안전하다.
- 호출 이벤트 발행 확인
- 도착 이벤트 발행 확인
- 실제 탑승/하차 확인
- passengerCount 변화 확인
- 단일 호출이 아니라 2~3개 연속 호출에서 queue 순서 확인
- direction reversal 시점 확인

### 2) NPC 일과 테스트는 “상태 존재”가 아니라 “순서 + 시간”으로 올리는 편이 좋다
최소한 다음 정도는 들어가는 편이 좋다.
- 시작 상태 확인
- `Working` 진입 시점이 스케줄 시간대와 대체로 맞는지
- `Resting` 전환이 work 종료 이후인지
- 하루 종료 시 home 관련 상태/위치로 돌아오는지
- 가능하면 층간 이동 중 waiting/in-elevator 흔적 확인

### 3) SimClock/EventBus는 count 외에 payload와 경계 이벤트를 같이 보는 편이 좋다
- `HourChanged` payload hour 값 확인
- `DayChanged` 경계 확인
- 이벤트 순서 확인
- pause / speed 변경 시 event count 변동 확인

### 4) 가능하면 Bootstrapper 기준 headless integration test가 하나는 있어야 한다
현재 테스트는 개별 시스템 조합을 직접 만드는 방식이다. 여기에 더해 실제 조립 루트에 가까운 테스트가 하나 있으면, 문서와 구현의 간극을 더 잘 잡아낼 수 있다.

## 최종 판단

### 최종 판정: 조건부 합격
이번 태스크는 **통합 테스트가 없던 상태에서 최소한의 통합 안전망을 만든 작업**으로는 충분히 의미가 있다.

하지만 보수적으로 보면, 아직 아래 수준에 머문다.
- 시스템들이 **대충 이어져 있다**는 확인
- 이벤트가 **아예 죽지 않았다**는 확인
- 상태가 **어느 시점엔가 바뀐다**는 확인

반면 아직 부족한 것은 이쪽이다.
- 설계 문서가 의도한 **정확한 계약 검증**
- 일정/수송/이벤트의 **순서와 시점** 검증
- 실제 게임 루프 기준 **상위 통합 정합성** 검증

따라서 이번 결과는 **“통합 테스트 도입 성공”으로는 인정 가능**하지만, **“설계대로 잘 되었음이 충분히 입증되었다”라고 보기는 어렵다.**

좀 더 직설적으로 말하면,
**이번 테스트 3개는 초록불을 하나 더 켠 것이지, 대시보드 전체가 안전하다는 증명은 아니다.**
