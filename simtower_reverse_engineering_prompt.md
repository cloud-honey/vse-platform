## ◆ 사전 지시 (AI에게)

당신은 1994년 고전 건설 경영 시뮬레이션 게임 **심타워(SimTower: The Vertical Empire / The Tower)**를 코드 수준으로 재현하기 위한 역분석 전문가입니다.

이 게임은 다음 특수성을 가집니다:
- 원개발: 일본 OPeNBooK / Yoot Saito → 맥시스가 미국판 퍼블리싱
- 한국판: "타워" / "타워 스페셜" 이름으로 다우기술 유통 (한글 더빙 포함)
- 플랫폼: Windows 3.1 (16bit NE 실행파일, WinG 그래픽 라이브러리 사용)
- 저장 포맷: `.TDT` (Tower DaTa) 바이너리 파일

**한국어, 영어, 일본어 자료를 모두 조사하여 각 언어 커뮤니티에서만 알려진 정보까지 수집**하고, 아래 각 항목을 최대한 수치와 알고리즘 수준으로 기술하세요.

## ■ 1. 게임 전체 구조 및 세계 규칙

### 1-1. 맵·공간 구조
- 타워 전체 폭: 몇 픽셀? 몇 타일? (확인된 값: 375세그먼트 = 3750픽셀?)
- 1타일(최소 단위)의 픽셀 크기 (가로×세로): 정확한 수치
- 지하층 수 최대, 지상층 수 최대 (B10~F110?)
- raw_floor 인덱스 → 실제 층수 변환 공식 (B10=raw0, 1F=raw10 등)
- 각 층에 배치 가능한 최대 유닛 수
- 건물 너비 제한: 플레이어가 조절 가능한가? 최대/최소 너비는?

### 1-2. 시간 시스템 (게임 내 달력)
- 1주 = 현실 몇 초? (시뮬 속도별 수치 포함)
- 1주 구성: 평일(WD) 2일 + 주말(WE) 1일 — 각 시간대 길이
- 1분기(Quarter) = 몇 주? 분기 결산 타이밍 정확히 언제?
- 시간대별 사람 이동 패턴 (출근 8~9시, 점심 12~1시, 퇴근 5~6시 등의 정확한 내부 틱 값)
- 연중 특수 이벤트 달력 (크리스마스 산타, VIP 방문 조건 등)
- 시뮬레이션 속도 단계 및 Fast Mode 동작 방식

### 1-3. 별점(Star Rating) 진행 시스템
- 각 별점 달성 조건 (인구 수 + 추가 조건) 전체 표:
  - ★1: 초기 상태
  - ★2: 300인 + [추가 조건?]
  - ★3: 1,000인 + [추가 조건?]
  - ★4: [인구?] + VIP 만족 숙박 + 메트로 스테이션 건설?
  - ★5: [인구?] + [조건?]
  - TOWER 등급: 15,000인 + 100층 성당 + 결혼식 개최
- 인구 카운트 방식: 누가 카운트되는가? (사무실 직원, 콘도 주민, 호텔 투숙객, 방문자 각각의 가중치)
- 별점 하락 조건: 있는가? 없는가?
- 각 별점에서 언락되는 시설 목록 (완전한 표)

## ■ 3. 엘리베이터 시뮬레이션 (핵심 시스템)

> 이 게임은 원래 엘리베이터 관리 시뮬레이터로 시작되었습니다. 가장 중요한 섹션입니다.

### 3-1. 하드웨어 제한값
- 최대 엘리베이터 샤프트 수: **24개** (모든 종류 합산)
- 샤프트당 최대 엘리베이터 카 수: **8개**
- 일반 엘리베이터 최대 서비스 층수: **30층**
- 고속 엘리베이터: 층수 무제한, 15층 배수(스카이 로비)에만 정차
- 서비스 엘리베이터: 최대 서비스 층수? 일반 엘리베이터와 구분되는가?
- TDT 파일 내 엘리베이터 데이터 구조: 샤프트당 194바이트 헤더 + 최대 29개×324바이트 + 최대 8개×346바이트

### 3-2. 엘리베이터 이동 알고리즘
- 기본 알고리즘 방식: SCAN 계열? LOOK 알고리즘? 다른 방식?
- 요청 처리 순서: 같은 방향 요청 우선? FIFO? 가장 가까운 층 우선?
- 여러 카가 같은 샤프트 공유 시 분배 로직
- 대기 중 정지층 설정 (시간대별 설정 가능한 스케줄): 내부 데이터 구조
  - `response_distances[14]`, `express_modes[14]`, `departure_delays[14]` — 이 14개 슬롯의 의미는?
- 엘리베이터 스케줄 타임슬롯 수: **14개** → 각 슬롯은 하루 몇 분 단위?
- 고속 엘리베이터 모드: Express to Top / Express to Bottom 모드 전환 조건

### 3-3. 입주자 이동 경로 탐색
- 사람이 목적지까지 경로를 어떻게 결정하는가? (BFS? 단순 그리디?)
- **엘리베이터 환승 횟수 제한: 최대 1회** — 이것이 경로 탐색에 어떻게 반영되는가?
- 계단 이용 최대 층수: **4층** — 엘리베이터 없이 4층 초과 이동 불가
- 스카이 로비 → 다음 엘리베이터 환승 로직
- 목적지까지 경로가 없을 경우 사람의 행동: 그냥 대기? 포기? 스트레스 급증?
- 사람이 이동할 때 픽셀 이동 속도 (층간, 횡방향)

### 3-4. 대기 시간 및 스트레스 연산
- 엘리베이터 대기 시작 → 스트레스 증가 시작까지의 허용 대기 시간 (초/틱)
- 스트레스 증가 속도 (단위시간당 증가량)
- 스트레스 색상 임계값: 노란색 → 분홍색 → 빨간색 각각의 수치
- 스트레스가 최대에 도달했을 때의 처리: 즉시 퇴거? 누적 카운트?
- 스트레스를 낮추는 조건 (엘리베이터 탑승 성공, 목적지 도착, 로비 대기 완화 등)
- "슈퍼 로비" (2층, 3층 로비) 가 스트레스를 줄이는 메커니즘: 구체적으로 어떻게 작동하는가?

## ■ 4. 테넌트 AI 및 행동 패턴

### 4-1. 사람(People) 오브젝트 데이터 구조
- TDT 파일 내 People 레코드 크기 및 필드 구성
- 각 사람이 보유하는 속성: 위치, 목적지, 스트레스 수치, 소속 유닛 ID, 현재 상태(대기/이동/근무/수면 등)
- 동시에 존재 가능한 최대 인원 수 (메모리/성능 제한)

### 4-2. 시설별 이동 패턴 (시간대 기반)
각 시설 이용자의 하루 동선을 완전히 기술하세요:

**사무실 직원:**
- 08:00~09:00 — 지상 로비 입장 → 해당 사무실로 이동
- 12:00~13:00 — 사무실 출발 → 패스트푸드/레스토랑 → 복귀
- 17:00~18:00 — 사무실 출발 → 지상 로비 퇴장
- 주말: 전혀 출근하지 않음
- 비오는 날 패스트푸드 이용률 변화 있는가?

**콘도 주민:**
- 평일/주말 외출 패턴의 차이
- 야간 귀가 시간대

**호텔 투숙객:**
- 체크인/체크아웃 시간
- 밤에 인구수 급증하는 메커니즘
- 청소(하우스키핑) 요구 조건 및 주기

**레스토랑/상점 방문자:**
- 레스토랑은 점심만 영업? 저녁 영업 없음? — 정확한 시간대
- 영화관 관객 유입 조건 및 타이밍

### 4-3. VIP 시스템
- VIP 방문 트리거 조건 (별점 3→4 전환 필요 조건)
- VIP 체류 시간 및 판정 기준
- VIP 스위트룸 요구사항: 주차장 필수? 거리 조건?
- VIP 판정 실패 시 페널티

### 4-4. 테넌트 퇴거 메커니즘
- 스트레스 외에 퇴거를 유발하는 조건
- 사무실 평가(Evaluation)가 빨간색으로 유지되는 기간 임계값
- 콘도 매매 시스템: 중고 콘도 가격 하락/복구 메커니즘 (가격 내렸다가 올리면 상태 복구되는 버그?)
- 퇴거 후 해당 유닛의 상태 변화

## ■ 6. 재난·이벤트 시스템

### 6-1. 화재(Fire)
- 발생 조건 및 확률 (별점에 따라 변화?)
- 확산 메커니즘 (층·유닛 단위?)
- 소방서 자동 진압 vs 보안실 진압 — 성공률 차이, 비용 차이
- 소방서 호출 비용 정확한 금액
- 화재 후 피해 처리 (유닛 파괴? 데미지?)

### 6-2. 폭탄 협박(Bomb Threat)
- 발생 조건 및 빈도
- 탐지 메커니즘: 보안실 반경과 탐지 확률
- 몸값 금액 (정확한 수치)
- 폭탄 폭발 시 피해 반경 및 처리

### 6-3. 바퀴벌레 감염(Cockroach)
- 발생 조건: 청소 안 된 호텔 방에서만?
- 확산 규칙 (인접 유닛으로 전파 속도)
- 제거 방법: 불도저로 철거만 가능?
- 방치 시 연쇄 확산 범위

### 6-4. 기타 이벤트
- 크리스마스 산타 이벤트: 날짜 조건, 시각적 효과만인가 게임 효과도 있는가?
- 결혼식 이벤트: 성당 건설 후 자동 발생? 조건은?
- 버스/메트로 방문객 유입 메커니즘

## ■ 7. 저장 파일(.TDT) 기술 구조

> 이 섹션은 재현 구현을 위한 데이터 구조 설계에 직접 활용됩니다.

### 7-1. 파일 헤더
- 전체 헤더 크기: ~490바이트?
- 두 번째 바이트 고정값: 0x24
- 파일 버전 정보 위치 및 포맷

### 7-2. Floor 레코드 구조
```
Floor_Struct:
  ULInt16  tenant_count        // 이 층의 유닛 수
  ULInt16  left_edge           // 층 왼쪽 경계
  ULInt16  right_edge          // 층 오른쪽 경계
  Array[tenant_count] Tenant_Struct
  Array[94] ULInt16            // tenant_id_to_index 리맵핑 테이블
```
- 188바이트 리맵핑 블록의 정확한 의미?
- 층당 최대 유닛: 94개 = (375세그먼트 / 4세그먼트)?

### 7-3. Tenant(유닛) 레코드 구조 (18바이트)
```
Tenant_Struct (18 bytes):
  ULInt16  left_edge
  ULInt16  right_edge
  ULInt8   type               // 유닛 종류 (타입 코드 전체 목록 필요)
  ULInt8   status
  ULInt8   pertype_idx
  Unknown  _unk7 (1byte)
  ULInt32  people_offset
  ULInt8   id
  Unknown  _unk13 (3bytes)
  ULInt8   rent_class
  Unknown  _unk17 (1byte)
```
- type 코드 전체 매핑 테이블 (사무실=?, 콘도=?, 패스트푸드=?, 엘리베이터=? 등)
- status 필드의 각 값 의미
- rent_class 필드의 각 값 의미

### 7-4. Elevator 레코드 구조
- 샤프트당 데이터: 194바이트 헤더 + (샤프트 높이 × 324바이트) + (카 수 × 346바이트)
- 헤더 내 필드: `left_edge`, `top_floor`, `bottom_floor`, `shaft_visible`, `response_distances[14]`, `express_modes[14]`, `departure_delays[14]`

## ■ 10. 버그 및 숨겨진 동작

### 10-1. 알려진 버그
- 콘도 가격 조작 버그: 퇴거 → 가격 인하 → 가격 인상 시 상태 복구 + 차익 발생 메커니즘
- 엘리베이터 비활성화 버그: 특정 층 정차 비활성화 시 대기열이 사라지는 현상
- "Tower는 왼쪽에서 오른쪽으로 인식한다" — 좌측/우측 비대칭 건설 시 빈 공간 처리 오류
- 레스토랑은 점심만 영업한다 — 의도된 설계인가, 버그인가?
- 사람들이 더 가까운 엘리베이터를 무시하고 이미 줄 선 엘리베이터에 계속 줄 서는 현상 (AI 한계)

### 10-2. 이스터에그
- 영화관 상영작 중 "Samurai Cop" 포함 — 다른 상영작 목록
- 치트: 특정 위치 클릭으로 시작 자금 2배
- 개발자가 숨긴 다른 메시지나 숨겨진 기능
- 2층/3층 로비 건설 단축키: Ctrl = 2층, Ctrl+Shift = 3층 — 다른 숨겨진 단축키

### 10-3. 버전 간 차이
- SimTower v1.0 vs v1.1 패치의 변경 내용
- SimTower (미국판) vs The Tower (일본판) vs 타워 스페셜 (한국판) 차이점:
  - 한국판에서 추가된 시설이나 변경된 수치
  - 일본판 v1.3이 SimTower보다 업그레이드된 부분
  - 한국판 더빙 음성 트리거 조건 전체 목록
